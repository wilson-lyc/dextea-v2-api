# 产品定制功能设计

## 概述

产品定制功能允许为每个产品配置若干「定制项」，每个定制项下有多个「定制选项」。顾客下单时，每个定制项**只能单选**一个选项（不支持多选）。

定制选项可以与原料库存绑定——例如「加芒果果粒」绑定芒果原料并设定消耗量；不需要消耗库存的选项（如温度、糖度）则无需绑定任何原料。

---

## 数据模型

### 实体关系

```
product (已有)
  └─ product_customization_item (1:N)  — 定制项，如"温度"
       └─ product_customization_option (1:N)  — 定制选项，如"热"、"少冰"
            └─ customization_option_ingredient (1:N, 可为空)  — 消耗原料绑定
```

### 表结构说明

| 表名 | 说明 |
|---|---|
| `product_customization_item` | 定制项：归属产品、名称、排序、是否必选 |
| `product_customization_option` | 定制选项：归属定制项、名称、价格调整、是否默认 |
| `customization_option_ingredient` | 选项与原料的绑定：选项ID、原料ID、消耗数量、单位 |

DDL 文件：`docs/sql/product_customization.sql`

### 设计要点

- **单选约束**在业务逻辑层保证，表结构不做强制（下单时服务端校验选项数量 = 1）。
- `is_default` 在同一定制项内仅允许一个选项为默认值，由服务层在写入时维护。
- `price_adjustment` 支持加价、减价和不变（0），便于促销/特殊配料的定价。
- `customization_option_ingredient` 与 `ingredient` 表通过 `ingredient_id` 关联，无外键约束（与现有原料模块保持一致）。
- 软删除：`status=0` 表示已删除，不做物理删除。

---

## 模块归属

功能在 **`product-service`** 模块中实现，复用该模块已有的原料（`ingredient`）表和产品（`product`）表。

## API 设计

所有接口均需登录鉴权（`@SaCheckLogin`）。

### 定制项管理

| 方法 | 路径 | 说明 |
|---|---|---|
| `POST` | `/v1/admin/products/{productId}/customization-items` | 新增定制项 |
| `GET` | `/v1/admin/products/{productId}/customization-items` | 查询产品的全部定制项（含选项） |
| `PUT` | `/v1/admin/customization-items/{itemId}` | 更新定制项 |
| `DELETE` | `/v1/admin/customization-items/{itemId}` | 删除定制项（软删除，联级软删除其选项及原料绑定） |

### 定制选项管理

| 方法 | 路径 | 说明 |
|---|---|---|
| `POST` | `/v1/admin/customization-items/{itemId}/options` | 新增定制选项 |
| `PUT` | `/v1/admin/customization-options/{optionId}` | 更新定制选项 |
| `DELETE` | `/v1/admin/customization-options/{optionId}` | 删除定制选项（软删除，联级清除原料绑定） |

### 原料绑定管理

| 方法 | 路径 | 说明 |
|---|---|---|
| `PUT` | `/v1/admin/customization-options/{optionId}/ingredient` | 绑定或更新原料（upsert） |
| `DELETE` | `/v1/admin/customization-options/{optionId}/ingredient` | 解绑原料 |

## 关键设计决策

### 1. 单选约束在服务层保证

`product_customization_item` 本身代表一个单选维度——数据库表结构中没有多选标志，业务语义就是「每项选一个」。下单时 order-service 应校验每个必选定制项有且仅有一个选项被选中。

### 2. 级联软删除

删除定制项时，服务层批量将其下所有选项标记为 `status=0`，并删除这些选项关联的 `customization_option_ingredient` 记录（物理删除绑定关系，因为它是纯关联表，无独立生命周期）。同理，删除选项时也清除其原料绑定。

### 3. 默认选项维护

`is_default` 在同一定制项内**最多一个**为 `true`。当某个选项被设置为默认时，服务层先将该定制项下所有选项的 `is_default` 置为 `false`，再将目标选项设为 `true`。使用 `@Transactional` 保证原子性。

### 4. 原料绑定 upsert（PUT 语义）

`PUT /customization-options/{optionId}/ingredient` 采用 upsert 逻辑：若绑定已存在则更新数量和单位，否则新增。利用 `UNIQUE KEY uq_option_ingredient` + MyBatis-Plus `saveOrUpdate` 实现。

### 5. 查询返回完整树形结构

`GET /v1/admin/products/{productId}/customization-items` 返回该产品下所有定制项及其选项、原料绑定的完整嵌套结构，避免前端多次请求。实现上：
1. 查询该产品下所有 `status=1` 的定制项。
2. 批量查询这些定制项下的所有选项。
3. 批量查询这些选项的原料绑定，并 JOIN `ingredient` 表获取原料名称。
4. 在内存中组装树形响应。

### 6. 不引入 XML Mapper

与现有代码库保持一致，全部使用 `LambdaQueryWrapper` 完成查询，不新增 XML 文件。

## 未来扩展方向

- **下单库存预扣**：order-service 下单时，根据所选定制选项的原料绑定，调用 `InventoryAdminService.adjustInventory` 扣减库存。
- **定制模板复用**：将常用定制项组合（如标准奶茶规格）抽象为模板，支持批量绑定到多个产品。
- **前台展示接口**：新增 `StoreBizController` 接口，按门店过滤掉库存为零的选项，向顾客端返回可用选项。
