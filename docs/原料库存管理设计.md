# Ingredient Inventory Management — Implementation Design

## Overview

Each store maintains its own stock of ingredients. This feature provides CRUD + stock-adjustment APIs for administrators to manage per-store ingredient inventory.

---

## Service Responsibility Split

| Module | Role |
|---|---|
| `product-service` | Owns the `store_ingredient_inventory` table; hosts all admin APIs and business logic |
| `store-service` | Provides an internal OpenFeign endpoint to validate store existence/status |
| `store-service-api` | Feign contract module (already exists) — exposes `StoreInternalFeign` |

---

## API Design

Base path: `POST /v1/admin/ingredient-inventories/stores/{storeId}`

All endpoints require an authenticated session (`@SaCheckLogin`).

| Method | Path | Description |
|---|---|---|
| `GET` | `/v1/admin/ingredient-inventories/stores/{storeId}` | Paginated list with optional filters |
| `GET` | `/v1/admin/ingredient-inventories/stores/{storeId}/{ingredientId}` | Single inventory detail |
| `PUT` | `/v1/admin/ingredient-inventories/stores/{storeId}/{ingredientId}` | Set inventory (upsert) |
| `POST` | `/v1/admin/ingredient-inventories/stores/{storeId}/{ingredientId}/adjust` | Adjust stock by delta |

### Query Filters (GET list)

| Parameter | Type | Description |
|---|---|---|
| `current` | Integer | Page number, default 1 |
| `size` | Integer | Page size, default 10, max 100 |
| `ingredientName` | String | Optional fuzzy match on ingredient name |
| `lowStock` | Boolean | When `true`, only return records where `quantity <= warnThreshold` |

### Set Inventory Request (`PUT`)

```json
{
  "quantity": 500.00,
  "unit": "g",
  "warnThreshold": 100.00
}
```

Calling this endpoint is always treated as a restock event — `lastRestockTime` is updated to now.

### Adjust Inventory Request (`POST .../adjust`)

```json
{
  "delta": -50.00,
  "reason": "制作订单 #12345"
}
```

- Positive `delta` → restock (updates `lastRestockTime`)
- Negative `delta` → consumption
- Returns `32006 INSUFFICIENT_STOCK` if the resulting quantity would go below zero

## Data Model

### Table: `store_ingredient_inventory`

```sql
CREATE TABLE store_ingredient_inventory (
    id               BIGINT       NOT NULL AUTO_INCREMENT,
    store_id         BIGINT       NOT NULL COMMENT '门店ID',
    ingredient_id    BIGINT       NOT NULL COMMENT '原料ID',
    quantity         DECIMAL(10,2) NOT NULL DEFAULT 0.00 COMMENT '当前库存量',
    unit             VARCHAR(16)  NOT NULL COMMENT '计量单位',
    warn_threshold   DECIMAL(10,2) NOT NULL DEFAULT 0.00 COMMENT '预警阈值',
    last_restock_time DATETIME    NULL COMMENT '最近一次补货时间',
    create_time      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    update_time      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    UNIQUE KEY uq_store_ingredient (store_id, ingredient_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='门店原料库存';
```

The `UNIQUE KEY uq_store_ingredient` guarantees one record per store-ingredient pair, preventing duplicate inserts under concurrent requests.

## Key Design Decisions

### 1. Upsert via `setInventory` (PUT)

Rather than separate Create and Update endpoints, a single `PUT /{ingredientId}` acts as an upsert. This matches the REST semantics of PUT (idempotent set) and simplifies the client.

### 2. Delta-based `adjustInventory` (POST)

Stock changes from order processing or manual corrections are expressed as signed deltas rather than absolute values. This is the standard pattern for inventory systems because:
- It naturally models the physical action (add 200 g, consume 50 g).
- It is safe under concurrent requests — a CAS/optimistic-lock upgrade can be layered on later.
- It keeps an audit-friendly interface (the `reason` field captures intent).

### 3. Store validation via Feign (`StoreInternalFeign`)

`setInventory` calls `store-service` through `StoreInternalFeign.checkStoreValidity` to ensure the target store exists and is not closed. Subsequent operations (adjust, detail, list) do not re-validate the store on every call — they rely on the inventory record already existing (which was validated at creation time). This avoids repeated cross-service calls on hot read paths.

### 4. Ingredient name filter using two-query strategy

Rather than writing a custom SQL JOIN, the page query uses two queries:
1. Resolve matching ingredient IDs by name (cheap, indexed).
2. Query inventory with an `IN (...)` clause on those IDs.

This is consistent with the existing codebase pattern (LambdaQueryWrapper only) and avoids XML mapper complexity. For very large datasets a native JOIN query would be more efficient, but this is sufficient at the current scale.

### 5. `lowStock` flag computed in-memory

The `lowStock` boolean in `InventoryDetailResponse` is computed as `quantity <= warnThreshold` at response construction time. The page-level `lowStock` filter is pushed down to the database with `.apply("quantity <= warn_threshold")` to avoid fetching unnecessary rows.

### 6. `lastRestockTime` semantics

Updated whenever stock increases — either through `setInventory` (always treated as a restock) or through a positive-delta `adjustInventory`. This allows operators to see how recently each ingredient was replenished without additional audit tables.

## Error Codes

| Code | Constant | Message |
|---|---|---|
| 32001 | `INVENTORY_NOT_FOUND` | 库存记录不存在 |
| 32002 | `STORE_NOT_FOUND` | 门店不存在或已关闭 |
| 32003 | `INGREDIENT_NOT_FOUND` | 原料不存在 |
| 32004 | `SET_FAILED` | 库存设置失败 |
| 32005 | `ADJUST_FAILED` | 库存调整失败 |
| 32006 | `INSUFFICIENT_STOCK` | 库存不足，调整后数量不能为负 |

## Future Improvements

- **Optimistic locking** on `adjustInventory` using a `version` column to eliminate lost-update races under high concurrency.
- **Inventory change log** table to record every delta adjustment with actor, timestamp, and reason for full audit trail.
- **Low-stock event publishing** (e.g. Spring ApplicationEvent or message queue) to trigger alerts or auto-replenishment workflows.
- **Batch set endpoint** (`PUT /v1/admin/ingredient-inventories/stores/{storeId}`) accepting a list, for efficient bulk initialization.
